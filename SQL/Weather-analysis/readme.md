# Анализ непрерывных периодов одинаковой погоды

---

## 🎯 Цель задания

Имея временную последовательность ежедневных наблюдений за погодой (солнечно, облачно, дождь и т.д.), необходимо определить **длину каждого непрерывного периода** с одинаковым типом погоды.

---

## 📊 Исходные данные

![Данные о погоде](https://github.com/user-attachments/assets/862b93a9-ea14-4ade-bd29-630f8f622fd9)

Таблица `weather` содержит:
- `the_date` — дата наблюдения  
- `weather_type` — тип погоды в этот день

---

## 🔧 Подход к решению

1. **Сравнение с предыдущим днём**  
   Используем оконную функцию `LAG()` для получения типа погоды за предыдущий день.

2. **Маркировка смены погоды**  
   Создаём вспомогательный флаг `difference`:
   - `1`, если погода **изменилась** по сравнению с предыдущим днём,  
   - `0`, если погода **осталась прежней** (или это первый день).

3. **Формирование идентификатора периода**  
   Вычисляем **кумулятивную сумму** флага `difference`. Каждое изменение погоды увеличивает эту сумму, что создаёт уникальный `period_id` для каждого непрерывного блока одинаковой погоды.

4. **Группировка и подсчёт**  
   Группируем записи по `period_id` и `weather_type`, чтобы получить длину каждого периода.

## 🧠 Почему это важно?

Это упражнение демонстрирует мощный паттерн анализа временных рядов в SQL:

- **Оконные функции**: `LAG()`, `SUM() OVER (...)`  
- **Условная логика**: `CASE WHEN … THEN … END`  
- **Динамическая сегментация**: создание виртуальных групп «на лету»  
- **Группировка по сформированным периодам**

Такой подход широко применяется в:

- **Анализе пользовательского поведения** — выделение сессий, периодов активности или бездействия  
- **Финансовом мониторинге** — определение фаз роста, стагнации или снижения показателей  
- **Промышленной диагностике** — измерение длительности состояний оборудования (включено/выключено, ошибка/норма)  
- **Data Science и бизнес-аналитике** — выявление устойчивых трендов, аномалий и циклических паттернов во временных данных
